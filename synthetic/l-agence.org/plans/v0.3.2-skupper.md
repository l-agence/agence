# Phase 5: v0.3.2+ — Skupper Multi-Cloud Distribution

**Status**: 📍 **SPEC ONLY** (After v0.3.1 proven stable)  
**Owner**: Sonnet + DevOps  
**Estimate**: TBD (design-dependent)

---

## 📌 Objective

Distribute local multi-agent swarm across clouds via Skupper. Reuse orchestrator logic, extend task queue to be geographically aware, maintain single control plane authority.

---

## 🎯 Conceptual Deliverables

- [ ] Skupper VAN setup (virtual application network)
- [ ] Task queue becomes geographically distributed (but logic unchanged)
- [ ] Agent containers run on any Skupper node (WSL-Ubuntu same everywhere)
- [ ] Matrix math + scheduling logic identical (just bigger graph)
- [ ] Append-only ledger for cross-cluster audit
- [ ] Central approval authority (humans gated at control plane only)

---

## 🔧 Architectural Principle

**Before Skupper (Local):**
```
┌─────────────────────────────┐
│ [Orchestrator + DWM Gate]   │ (control plane)
├─ Agent: ralph              │
├─ Agent: aider              │
├─ Agent: sonny              │
└─ Task Queue + Matrix Math  │
```

**After Skupper (Distributed):**
```
┌─────────────────────────────┐     ┌─────────────────────────────┐
│ [Orchestrator + DWM Gate]   │     │  Skupper Site: eu-north     │
│ (control plane, single)     │────→├─ Agent: ralph-eu            │
├─ Task Queue (global)       │     ├─ Agent: aider-eu            │
├─ Append-Only Ledger        │     └─ (sees global queue)         │
└─────────────────────────────┘     
                ↑
                │ (task assignment)
                ↓
        ┌─────────────────────────────┐
        │  Skupper Site: us-west      │
        ├─ Agent: sonny-us            │
        ├─ Agent: ralph-us            │
        └─ (sees global queue)        │
                └─────────────────────┘
```

**Invariants**:
- ✅ One orchestrator (not replicated)
- ✅ One task queue (not replicated)
- ✅ One approval authority (not replicated)
- ✅ Agent containers identical on every node (no special logic)

---

## 🎓 Design Decisions (Deferred)

- How to handle latency between sites? (task assignment → execution)
- What's the failure mode if a site disconnects? (task reassignment?)
- How to consolidate DWM lessons across sites? (geo-dup detection?)
- Append-only ledger: Git? Or separate commitment log?

---

## 📁 Likely Modifications

- [ ] Orchestrator: geo-aware task assignment (prefer local agents)
- [ ] Task queue: includes site hint (but not binding)
- [ ] Append-only ledger: Git branch per site for conflict-free merges
- [ ] Skupper: service exposure (queue, orchestrator endpoints)
- [ ] Tests: simulate site disconnect, network latency

---

## ⚠️ Critical Constraint

**Skupper is just distribution.** The hard problems are architecture:
1. Orchestrator doesn't need to be replicated (gossip or centralized)
2. Task queue doesn't need to be replicated (Git's merge strategy handles conflicts)
3. Agent code is identical everywhere (only environment differs)

**Result**: Skupper is a network tool, not an architectural shift.
